# 룰 vs 스킬 vs 에이전트 — 선택 가이드

## 한 줄 정의

| 구분 | 정의 | 비유 |
|------|------|------|
| **룰** | Claude가 항상 따르는 행동 원칙 | 사규/규정집 |
| **스킬** | 사용자가 트리거하는 절차 레시피 | 업무 매뉴얼 |
| **에이전트** | 독립 컨텍스트에서 실행되는 전문 인스턴스 | 전문 직원 |

---

## 핵심 차이: 언제 적용되는가

```
룰       → 항상, 자동, 묵시적
스킬     → 사용자가 명시적으로 호출할 때
에이전트 → 위임이 필요한 특정 시점에
```

---

## 상세 비교

| 항목 | 룰 | 스킬 | 에이전트 |
|------|-----|------|---------|
| 트리거 | 항상 적용 | `/명령어` 또는 요청 | 스킬/훅/Claude 판단 |
| 실행 주체 | Claude 행동 제약 | 메인 Claude | 별도 인스턴스 |
| 컨텍스트 | 공유 | 공유 | 독립 분리 |
| 병렬 실행 | - | 불가 | 가능 |
| 용도 | 일관성 보장 | 절차 실행 | 전문 분석/검증 |
| 수정 효과 | 즉시 전체 적용 | 해당 워크플로우만 | 해당 에이전트만 |

---

## 실전 선택 기준

```
"이게 항상 지켜져야 하는 원칙인가?"      → 룰
"사용자가 명령으로 실행하는 절차인가?"    → 스킬
"독립적 판단/분석/검증이 필요한가?"      → 에이전트
```

### 스킬 선택 체크리스트
- [ ] 사용자가 직접 시작하는 작업인가?
- [ ] 순서가 고정된 절차인가?
- [ ] 중간에 사용자 확인이 필요한가?
- [ ] 메인 컨텍스트의 상태(파일 경로, 변수)를 그대로 써야 하는가?

### 에이전트 선택 체크리스트
- [ ] 결과가 크고 메인 컨텍스트를 오염시킬 것 같은가?
- [ ] 다른 작업과 동시에(병렬로) 실행 가능한가?
- [ ] 특정 도메인 전문 지식이 필요한가?
- [ ] 훅으로 자동 트리거되어야 하는가?
- [ ] 독립적인 second opinion이 필요한가?

## 이 프로젝트 적용 예시

```
룰 (azure.md)
  "하드코딩 금지 — 모든 값은 variables.tf에"
  → tf 파일 작성할 때마다 자동으로 Claude가 준수

룰 (terraform.md)
  "항상 fmt → checkov → plan 순서로"
  → /tf-plan 스킬 없이 plan 요청해도 이 순서 따름

스킬 (/tf-plan)
  "fmt → checkov → plan → 비용분석을 순서대로 실행"
  → 사용자가 /tf-plan 칠 때만 실행

에이전트 (terraform-reviewer)
  "tf 파일 변경 후 코드 리뷰"
  → 훅이 자동 트리거, 독립적으로 리뷰 후 결과 반환
```

### 스킬이 에이전트를 호출하는 구조

```
/tf-apply (스킬)               ← 사용자가 직접 트리거, 절차 고정
    └→ plan-validator (에이전트) ← 결과가 크고 독립적 판단 필요
    └→ azure-validator (에이전트) ← 배포 후 검증, 병렬 실행 가능
    └→ cost-optimizer (에이전트) ← 도메인 전문성 필요
```

스킬이 "지휘자", 에이전트가 "연주자" 역할을 하는 구조.

## 잘못 배치하면 생기는 문제

| 잘못된 배치 | 문제 |
|------------|------|
| 룰에 절차를 넣음 | 모든 대화에서 해당 절차 강제 실행 → 과도한 동작 |
| 스킬에 원칙을 넣음 | `/명령어` 안 치면 원칙 무시됨 → 일관성 깨짐 |
| 에이전트로 간단한 절차 처리 | 오버엔지니어링, 컨텍스트 전달 비용 낭비 |
| 스킬로 전문 분석 처리 | 메인 컨텍스트 오염, 병렬 실행 불가 |
